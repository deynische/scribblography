Using a Custom Domain With GitHub Pages

14 Nov 2018 ｜ 7 min read
web / domains

There’s something irritating about turning to Google for help and finding articles written half a decade ago. Depending on what you’re looking for, either the advice is still sound or it’s incredibly outdated.

This made Scribblography’s launch unnecessarily complicated last week. I’m using GitHub Pages to host this blog but old advice made the custom domain set-up confusing. Now that everything is done, it’s clear to me the process is as streamlined as ever.

The usual GitHub Pages domain is something along the lines of:
your-username.github.io/project-repo-name

If you have your own domain, you can use it instead of the default domain. This involves two parts: CNAME records and A records.

Giving Your Domain A Direction

Think of a domain as the name of a place. For example: The Central Library. Just telling you about The Central Library doesn’t let you know where it is.

You can think of A records as the address to a place. I could say The Central Library is located on Fourth & Madison. Or I could say The Central Library is located at 1000 4th Ave. Either way, it points you to where you should go.

In terms of domains, A records are usually in the form of IP addresses. The A records need to be setup where ever your domain name is managed. This may or may not be the same place where your domain name is registered.

To use a custom domain with your GitHub Pages site, you need to set up A records pointing to GitHub’s IP addresses. As of this post, these are the four addresses:

  • 185.199.108.153
  • 185.199.109.153
  • 185.199.110.153
  • 185.199.111.153

Truthfully, you could probably get away with just one A record. But it doesn’t hurt to have all four.

Associating Your Domain With Your Project

Now that you have your domain pointing to GitHub, you need to set up the name association.

For instance, I told you The Central Library is located at 1000 4th Ave. But when you show up to that location, how do you know you’re at The Central Library? I don’t know about you, but I start looking for a sign on a door or a window.

You can think of CNAME records as way to identify your domain.

If you set up your custom domain in your GitHub project’s settings, it should automatically create a CNAME file for you within your repo branch.

If you have a set-up where you’re deploying your website updates to your repo branch, remember to include your CNAME file in your source folder. Otherwise, it might go missing and your project will lose the association to the domain name. Like a building missing it’s signage.

At this point, you should be all set up and ready to go!

Getting It All To Go To The Same Place

Across the web, you might have noticed how same places use the www prefix and others don’t. You might have also noticed how some places will automatically fill in the www whereas others remove it.

What’s going on?

The technical jargon is that www.example.com is an alias of example.com or vice versa.

It’s kind of like the how The Central Library around here is also known as The Seattle Public Library. If I say either name, you’ll know it’s the geometric building on Fourth & Madison. However, if you look it up, it’s technically The Seattle Public Library. It just happens to also be known as The Central Library.

Two different names. Same place.

For domains, you need to know this isn’t an automatic association. Whatever domain name you went with in your repo’s CNAME file is the proper name for your site. The other one probably doesn’t point to anything.

If you thought setting up another CNAME record is the solution, you’re right! However, you can’t have more than domain name in a CNAME record and you can’t have more than one CNAME file in the same place. That’s just too confusing.

To set up an alias, you need to go to where you set up your A records. From there, you set up a CNAME record to associate your alias domain to your GitHub account. It’ll look something like this:

Record Domain Data
CNAME www.example.com your-username.github.io

From here, GitHub Pages will automatically resolve your site domain to the version you want people to go to.

What About HTTPS?

I’m sure you’ve noticed that for a better secure web experience, HTTPS is now the standard. The S stands for “secure” and this is achieved through a secure sockets layer, known as SSL or an SSL certificate. In order to earn your S in HTTPS, you need to get your hands on one of these.

As of May 2018, you don’t need to do anything about this for your custom domain GitHub Pages site. GitHub has partnered with Let’s Encypt to help keep your Pages sites secure.

Really, you don’t need to do anything else.

What If I Really Need SSL?

For a long time, GitHub didn’t offer SSL to custom domains. There are still a plethora of articles instructing how to get SSL to work with GitHub Pages.

One popular method is to sign up with Cloudflare. They allow one free SSL certificate good for one site with your account. All you have to do to move your domain management to Cloudflare.

A Note On DNS Propagation

For any updates and changes to your domain records, it may be a while for them to take effect. Most places will tell you to wait a day or two. But if you’re worried, there’s a way to check how things are progressing without refreshing your browser a million times. You just need to get to your command line.

For Unix and iOS users, you’ll be using the dig command. It stands for domain information groper. For anyone working with networking, this is probably the preferred method.

For Windows users, you’ll be using the nslookup command. It stands for nameserver lookup.

Either way, you can use these commands to check your domain name and see where it points to if it’s pointing to anywhere. It’ll help your impatience feel productive while you wait. It did for me when I was navigating through all of this last week.
